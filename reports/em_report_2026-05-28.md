## 5. Prioritized Recommendations

Ordered by business impact. Each item names the *why* in plain language.

### P0 — Restart the project or formally pause it
**Symptom:** Zero commits since 2026-04-07; two CRITICAL alerts ignored for ≈ 7 weeks; single contributor.
**Action:** Decide explicitly whether this project is **active**, **on hold**, or **archived**. If active, assign an owner and a target milestone this week. If on hold, mark the repo `Archived` so stakeholders stop expecting movement.
**Why it matters:** "Looks alive but isn't" is the worst state — it blocks decisions and erodes trust.

### P1 — Triage the two open CRITICAL issues
**Symptom:** Issues #7 and #8 are unassigned and unanswered.
**Action:** Assign an owner today. Confirm whether the underlying incidents are real or simulated. If real, follow the agent's rollback/pool-size recommendations. If simulated, close the issues with a note so the queue is clean.
**Why it matters:** Critical-labelled issues that sit forever train the team to ignore the label.

### P2 — Introduce a real review gate
**Symptom:** 5/5 merged PRs had no human reviewer; the same person opens and merges every PR.
**Action:** Add a branch-protection rule on `main` requiring at least one approving review from someone other than the author. If staffing is only one engineer, treat the AI risk-analysis workflow output as a *required check* (block merge on `REQUEST CHANGES`) until a second human is available.
**Why it matters:** Self-merge without review is the single largest preventable source of regressions and is also a compliance/audit concern.

### P3 — Fix the three highest-impact known bugs
**Symptom:** Bugs that can produce wrong business numbers are still in `main`.
**Action:** Schedule fixes (with tests) for, in order:
1. **BUG #13** — stock can go negative (`StockMovementService.java:81`)
2. **BUG #19** — frontend "delete supplier" sends GET instead of DELETE (`frontend/src/services/api.js:16`)
3. **BUG #1 / #15** — purchase-order `lineTotal` not recalculated / not set on save
**Why it matters:** These three directly cause *wrong inventory counts* and *broken user actions* — the kind of defects an EM hears about from sales or finance, not from engineering.

### P4 — Convert the 9 TODO test cases into real tests
**Symptom:** Each TODO is explicitly tied to a known bug; writing them would simultaneously raise coverage and lock in fixes.
**Action:** Treat the 9 `// TODO: add test for …` comments as a single deliverable. Pair each with the corresponding fix from P3 / future bug work.
**Why it matters:** This is the highest-ROI testing work available — the targets are pre-specified and the harness is already wired.

### P5 — Reduce bus-factor and document operating norms
**Symptom:** One contributor, no LICENSE, no CONTRIBUTING, no on-call definition for the observability alerts.
**Action:** Add a second reviewer/contributor, add a `LICENSE` file, add a brief `CONTRIBUTING.md` describing the PR + review expectations and what to do when an observability issue is filed.
**Why it matters:** Without these, the project cannot safely accept outside help and cannot meaningfully respond to incidents.

### P6 — Close out the documentation backlog
**Symptom:** 40+ `// TODO: add Javadoc` markers on public APIs.
**Action:** Lower-priority cleanup; consider doing it in a single sweep, or relax the convention if Javadoc isn't actually a project requirement (in which case, delete the TODOs rather than carry them).
**Why it matters:** A repository full of "TODO" comments signals abandoned intent and inflates the apparent backlog.
